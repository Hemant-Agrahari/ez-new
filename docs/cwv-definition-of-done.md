# CWV Definition of Done (per page)

Every page is "done" only when it passes this checklist. Run through it before marking a page
shipped, before opening a PR, and before closing any CWV ticket. Until enforcement is automated,
this is the gate.

## The checklist

### Head order
- [ ] `<head>` matches the canonical order in `conventions/head-order.md`
- [ ] JSON-LD schemas appear **after** font preload + CSS, not before
- [ ] Analytics scripts (if any) are last in `<head>` and use `async`

### CSS
- [ ] Critical CSS is inlined in `<style>` and is under **14 KB gzipped**
- [ ] Exactly **one** `<link rel="stylesheet">` in the critical path (the page's `*.purged.css`)
- [ ] Bootstrap is included in the purged bundle, not loaded separately
- [ ] All other CSS uses `media="print" onload="this.media='all'"` (or is absent)
- [ ] `<noscript>` fallback exists for deferred CSS
- [ ] No inline `style="..."` in page body
- [ ] PurgeCSS safelist includes every class JS toggles on this page

### JS
- [ ] Every `<script src>` in `<head>` has `defer` or `async`
- [ ] No inline `<script>` blocks except the JSON-LD generated by `build.js`
- [ ] `build.js` `detectJsModules()` picks up every module this page needs
- [ ] No third-party CDN script references

### Images
- [ ] LCP element identified via Playwright (mandatory — see "LCP detection" below)
- [ ] If LCP is an `<img>` element → `lcp_image:` declared in frontmatter
- [ ] If LCP is a CSS background → `lcp_bg_mobile:` and/or `lcp_bg_desktop:` declared
- [ ] LCP preload emitted in `<head>` with `fetchpriority="high"`
- [ ] For `<img>` LCP: matching `<img>` has `fetchpriority="high"` + `loading="eager"`
- [ ] Exactly **one** image has `fetchpriority="high"`
- [ ] All above-fold images: `loading="eager"`
- [ ] All below-fold images: `loading="lazy"`
- [ ] Every `<img>` has `width` + `height` + `alt` + `decoding="async"`
- [ ] No `data-aos` on any above-the-fold element

### CSS backgrounds (added April 2026 after LCP 3.5 s → 1.7 s breakthrough)

- [ ] **No PNG or JPG in any `background-image:` declaration** — WebP only
- [ ] **No cross-origin URLs** in `background-image:` (e.g. `url(https://other-domain.com/...)`)
- [ ] If a CSS `background-image:` is the LCP on any viewport, it MUST have `lcp_bg_mobile:` or `lcp_bg_desktop:` frontmatter so it's preloaded
- [ ] Run `node tools/convert-bg-to-webp.js` after adding any new CSS background
- [ ] Run `node tools/rewrite-bg-css.js` to update references after conversion
- [ ] Re-run PurgeCSS + critical CSS extraction after CSS source changes

### LCP detection workflow (mandatory for new pages + page edits)

Every new page and every page content edit must run this:

Run `node tools/cwv-bench.js <page>` for LCP timing (Playwright-based, mobile 4G throttled).
To identify the **LCP element** (tag, URL, whether img src or CSS background), add a
PerformanceObserver snippet inline in a Playwright session. At 360×640 (mobile) and
1440×900 (desktop), capture:

```js
new PerformanceObserver((list) => {
  const entries = list.getEntries();
  const lcp = entries[entries.length - 1];
  console.log({
    element: lcp.element?.tagName,
    url: lcp.url,                           // set for image LCPs
    textContent: lcp.element?.textContent,  // set for text LCPs
    isBackground: lcp.url && !lcp.element?.src,  // CSS background-image LCPs
  });
}).observe({ type: 'largest-contentful-paint', buffered: true });
```

Decision tree based on what's reported:

| LCP type | Action |
|---|---|
| `<img>` element with URL | Set `lcp_image: /path/to/image.webp` in frontmatter |
| SECTION/DIV with CSS `background-image` URL | Set `lcp_bg_mobile:` and/or `lcp_bg_desktop:` |
| Text element (P, H1, DIV with text) | No image preload needed — font preload handles it |

**If the detected LCP URL ends in `.png` or `.jpg`**, convert it to WebP first via
`tools/convert-bg-to-webp.js` before setting the frontmatter.

### Measured metrics (Lighthouse mobile, throttled)
- [ ] Performance score ≥ **90**
- [ ] LCP ≤ **2.5 s**
- [ ] CLS ≤ **0.1**
- [ ] TBT ≤ **200 ms** (proxy for INP)
- [ ] FCP ≤ **1.8 s**

### Critical-CSS self-test (catches the mobile-menu trap)

Whenever `*.critical.css` changes or the extractor is modified, run this check in Playwright
at 375×812 **on every page touched**, not just one:

- [ ] Load with **all deferred CSS blocked** (purged/slick/aos/fa-subset routes aborted)
- [ ] `<header>` height ≈ 64 px (not expanded mobile menu)
- [ ] `.main_menu` computed `display` === `none` on mobile
- [ ] `<main>` `offsetTop` ≈ 64 (immediately below header, not pushed down by expanded menu)
- [ ] Reload normally, measure CLS via `PerformanceObserver({ type: 'layout-shift' })`
- [ ] CLS < 0.1 (ideally < 0.02)
- [ ] If pre-deferred layout differs from post-deferred layout, critical CSS is incomplete — DO NOT SHIP

Why mandatory: the April 2026 incident caused CLS 0.9483 live because this check didn't exist.
The extractor silently skipped mobile-menu hide rules and nobody noticed until PageSpeed showed
catastrophic CLS. See `docs/conventions/css-strategy.md` § "Critical CSS extraction — pitfalls".

### Functional smoke
- [ ] Page renders correctly with JS disabled (nav static, content visible)
- [ ] No console errors on load
- [ ] All forms submit successfully
- [ ] All modals open/close without layout shift
- [ ] All in-page links and CTAs go to the intended target

## How to run the metrics

```bash
# Build dist/
node build.js --page=white-label-igaming-platform

# Serve locally
node server.js   # (or: npx serve dist)

# Run Lighthouse against http://localhost:PORT/white-label-igaming-platform
# Mobile preset, throttled 4G, 4× CPU slowdown
```

Record the four metrics (Perf, LCP, CLS, TBT) in the page's commit message or PR description.

## Enforcement status

**Manual for now.** After conventions are validated on 2–3 pages and proven correct, this
checklist becomes a build-step validator and/or pre-commit hook. Until then, the checklist
lives here and agents + humans tick it by hand.

## What happens when a page fails

1. Identify the first failing item (ordering matters — head order failures cascade into metric failures).
2. Fix it — either in the page, the component, `build.js`, or the convention doc if the convention is genuinely wrong.
3. Rebuild the page, re-run Lighthouse, re-walk the checklist from the top.
4. Do not ship with any red checkbox.
